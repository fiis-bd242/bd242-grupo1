# 4.7. Módulo de Gestión de Comercios

### Caso de Uso 1: Registrar Comercio

| **Tarea de usuario realizada**  | El comerciante registra los datos de su comercio en el sistema. |
|-------------|-------------|
| **Actores Iniciadores**   | Comerciante |
| **Actores Participantes**     | Comerciante <br> -Área Comercial |
| **Flujo de Eventos** | 1. El comerciante accede al sistema.<br> 2. Ingresa los datos de su comercio.<br> 3. El sistema confirma la recepción de los datos. |
| **Excepciones**     | - Si los datos están incompletos, el sistema muestra un mensaje de error. |
| **Precondiciones**      | El comerciante debe tener acceso al sistema para registrarse. |
| **Postcondiciones**        | El comercio queda registrado en el sistema y el área comercial puede iniciar la validación de los datos. |
| **Incluye Casos de Uso**     | Validación de datos |
| **Servicios Utilizados**     | Sistema de gestión de comercios |
| **Requerimientos No Funcionales**   | - Seguridad: El acceso debe estar restringido a comerciantes registrados. <br> - Usabilidad: La interfaz debe ser intuitiva para facilitar el registro de información. |

---

### Caso de Uso 2: Validar Datos del Comercio

| **Tarea de usuario realizada**  | El área comercial valida la información proporcionada por el comerciante. |
|-------------|-------------|
| **Actores Iniciadores**   | Sistema |
| **Actores Participantes**     | Área Comercial |
| **Flujo de Eventos** | 1. El sistema envía los datos del comercio al área comercial.<br> 2. El área comercial valida los documentos y datos proporcionados.<br> 3. Si la validación es correcta, el sistema actualiza el estado del comercio a "validado". |
| **Excepciones**     | - Si la validación falla, el sistema alerta al comerciante para corregir los datos o remitir documentos faltantes. |
| **Precondiciones**      | El comercio debe haber completado el registro. |
| **Postcondiciones**        | Los datos del comercio son validados y se puede continuar con el proceso de aprobación. |
| **Incluye Casos de Uso**     | Atención personalizada |
| **Servicios Utilizados**     | Sistema de gestión de registros |
| **Requerimientos No Funcionales**   | - Seguridad: Solo el área comercial puede acceder a los datos sensibles del comercio. <br> - Disponibilidad: El sistema debe estar disponible para realizar la validación en cualquier momento. |

---

### Caso de Uso 3: Brindar Atención Personalizada al Comercio

| **Tarea de usuario realizada**  | El área de soporte ofrece atención personalizada al comercio si hay problemas en la validación o aprobación. |
|-------------|-------------|
| **Actores Iniciadores**   | Sistema |
| **Actores Participantes**     | Comerciante <br> -Área de Soporte |
| **Flujo de Eventos** | 1. El sistema detecta que los datos del comercio no pasaron la validación.<br> 2. El área de soporte se comunica con el comerciante para ofrecer asesoramiento personalizado.<br> 3. El comerciante recibe ayuda para completar los datos necesarios. |
| **Excepciones**     | - Si el comerciante no responde a la comunicación, el sistema reprograma el contacto o envía recordatorios. |
| **Precondiciones**      | El comerciante debe haber fallado en la validación o aprobación de datos. |
| **Postcondiciones**        | El comerciante recibe asesoría personalizada y puede completar su registro exitosamente. |
| **Incluye Casos de Uso**     | Validación de datos |
| **Servicios Utilizados**     | Sistema de notificaciones <br> -Sistema de Soporte|
| **Requerimientos No Funcionales**   | - Usabilidad: La interacción debe ser clara y fluida entre el soporte y el comerciante. |

---

### Caso de Uso 4: Activar Métodos de Pago

| **Tarea de usuario realizada**  | El área comercial activa los métodos de pago del comercio. |
|-------------|-------------|
| **Actores Iniciadores**   | Área Comercial |
| **Actores Participantes**     | Área Comercial |
| **Flujo de Eventos** | 1. El área comercial selecciona y activa los métodos de pago habilitados para el comercio.<br> 2. El sistema confirma la activación y lo comunica al comerciante. |
| **Excepciones**     | - Si ocurre un error en la activación, el sistema alerta al área comercial. |
| **Precondiciones**      | El comercio debe estar aprobado en la plataforma. |
| **Postcondiciones**        | Los métodos de pago están activos y el comercio puede comenzar a recibir pagos. |
| **Incluye Casos de Uso**     | Aprobación de registro |
| **Servicios Utilizados**     | Sistema de gestión de comercios |
| **Requerimientos No Funcionales**   | - Disponibilidad: El sistema de pagos debe estar disponible para la activación inmediata. |

---

### Caso de Uso 5: Capacitar Comercio

| **Tarea de usuario realizada**  | El área de soporte capacita al comercio en el uso de la plataforma. |
|-------------|-------------|
| **Actores Iniciadores**   | Área de Soporte |
| **Actores Participantes**     | Comerciante <br> -Área de Soporte |
| **Flujo de Eventos** | 1. El área de soporte programa una capacitación para el comerciante.<br> 2. El comerciante asiste a la capacitación.<br> 3. El sistema confirma que el comerciante ha completado la capacitación. |
| **Excepciones**     | - Si el comerciante no asiste, se reprograma la capacitación. |
| **Precondiciones**      | El comercio debe tener los métodos de pago activados. |
| **Postcondiciones**        | El comerciante está capacitado para operar en la plataforma. |
| **Incluye Casos de Uso**     | Activar Métodos de Pago |
| **Servicios Utilizados**     | Sistema de gestión de capacitaciones |
| **Requerimientos No Funcionales**   | - Disponibilidad: El sistema de capacitación debe estar disponible en horarios flexibles. |

---

### Caso de Uso 6: Brindar Retroalimentación

| **Tarea de usuario realizada**  | El sistema solicita y recopila retroalimentación del comerciante mediante una encuesta. |
|-------------|-------------|
| **Actores Iniciadores**   | Sistema |
| **Actores Participantes**     | Comerciante |
| **Flujo de Eventos** | 1. El sistema envía una encuesta al comerciante.<br> 2. El comerciante completa la encuesta.<br> 3. El sistema guarda la retroalimentación para su análisis. |
| **Excepciones**     | - Si el comerciante no responde, se envía un recordatorio. |
| **Precondiciones**      | El comerciante debe estar registrado y operativo en la plataforma por un periodo de tiempo |
| **Postcondiciones**        | El sistema almacena la retroalimentación para posibles mejoras. |
| **Incluye Casos de Uso**     | Aprobación de registro |
| **Servicios Utilizados**     | Sistema de encuestas |
| **Requerimientos No Funcionales**   | - Usabilidad: La encuesta debe ser fácil de completar y accesible en cualquier dispositivo. |
